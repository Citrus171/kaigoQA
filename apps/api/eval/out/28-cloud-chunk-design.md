## 考察

- アプローチCは部分有効: 8件中 3 件救済。残り 5 件は retrieval で src_id 自体が引けていない（1つも chunk が top-20 に入らない）か、generation が情報を使いこなせていない。
- 残る retrieval failure: 6/41 = 14.6%

- **generation(reasoning failure) が依然として主要律速**（8/41）。次の一手は generation 改善。

## ⚠️ 2026-06-19 検算: 本結果は無効（生成入力の非対称アーティファクト）

数値自体は再現する（C good=27/41=65.9%, 2×2 both19/ret6/odd8/rea8）。**しかし「oracle 61.0% を +4.9pt 超え」は評価アーティファクトで、retrieval 改善ではない。**

- approach C 実装（L188 `parent_document_refs`）は **top-20 に出現した全 src_id の親文書**を生成に渡す。top-20 には 14〜19 質問がヒット（≒検索が無差別）。
- 結果、生成入力は **平均 69.7 refs（53-89）/ 14.6 質問分**。oracle（その質問の M=平均4.9 refs）の **約14.3倍**。
- oracle 超えは「oracle は5件で生成、C は70件で生成」という**生成入力規模の非対称**が原因。retrieval 品質を測れていない。
- odd 8件（gold-A-002,010,026,029,032,033,037,115）は全件 parent_refs 57-88件。他質問 ref の大量混入で偶然 sufficient を満たした＝非対称の産物。
- retrieval failure 8→6 の改善も 70件注入の副作用で、chunk 設計の純粋効果と分離不能。
- **実運用非現実的**: 70件/質問はコスト・レイテンシ・ノイズで破綻。

### 正しい再実装（out/29 で実施）
- parent-document retrieval は「**検索ヒット chunk の親だけ（上位1〜3 src_id に限定）**」を渡す。
- oracle との公正比較には生成入力規模を oracle と同等オーダーに揃える。
- 鉄則: **oracle（理論上界）を超えたら、まず評価バグ（oracleと条件の非対称）を疑う**。out/26（judge非対称）に続く同型ミス。
