## 構造的限界

offload率をこれ以上上げられない根本原因: **edge 回答の絶対品質が cloud より 5.4pt 低い**。

| ポリシー | good率 | 備考 |
|---|---|---|
| always-edge | 10/34 = 29.4% | gemma3:4b の品質上限 |
| always-cloud | 8/23 = 34.8% | deepseek-v4-pro の品質 |
| 差 | **5.4pt** | この差が loss の下限を決める |

offload率を上げると routed の回答に edge が混ざる比率が増え、routed good率が always-edge 側に引っ張られる。quality loss < 5pt を維持できる offload の理論上限は「edge品質がcloud品質に近いほど」上がる。

現状の gemma3:4b では:
- 安全な offload: **最大 20.6%（7/34）**
- それ以上は loss が閾値を超えるか、不安定になる
- offload率を 50%+ にしたいなら、edge モデルの品質向上（gemma3:4b→granite3.2:8b 等）が必須

## 結論

1. **offload率は現行 14.7% から 20.6% まで安全に引き上げ可能**（loss 1.4pt < 5pt）
2. **推奨閾値: −0.0104〜0.0032**。offload 6〜7件（17.6%〜20.6%）
3. **20.6% を超える offload は edge 品質がボトルネック**。gemma3:4b の絶対品質（29.4%）が cloud（34.8%）に 5.4pt 劣るため
4. **コスト削減を最大化するには edge モデル自体の品質向上が必要**。`OLLAMA_GEN_MODEL=granite3.2:2b` で再評価し、edge good率が改善すれば offload 上限も上がる
5. **「ルーターで品質維持しながら offload 率を上げる」という当初目標は、edge 品質が真のボトルネックであることを示す結果**。ルーターの調整だけでは限界があり、SLM 選定自体が次の意思決定になる
